Architecture & User Manual

A guide to the internal workings and operational use cases of the BSP Kafka GC Analyzer.

System Overview

The BSP Kafka GC Analyzer is a lightweight, real-time telemetry monitoring and diagnostic web application tailored to investigate JVM Garbage Collection behaviors inside Apache Kafka deployments. The tool automates log retrieval across multi-cluster environments, calculates key performance statistics, detects GC anomalies (such as memory leaks and GC storms), and suggests configuration remedies.

By focusing specifically on GC metrics (rather than general VM metrics), it isolates issues related to long Stop-the-world (STW) pauses, throughput degradation, heap occupancy peaks, and allocation rates, allowing administrators to optimize JVM tunings.

Application Control

Operators should use manage-app.sh as the single control surface for local setup, deployment, startup, shutdown, status checks, and log access. It creates or repairs the Python virtual environment, installs dependencies, seeds demo/history data when the selected database is empty, starts the FastAPI server in the background, and waits for /api/health before reporting the app as started.

Primary Commands

Command Purpose
./manage-app.sh deploy 8083 Prepare the environment, seed data if needed, stop any existing process on the port, and start a fresh app process.
./manage-app.sh start 8083 --open Start the app in the background and open the local dashboard after the health check succeeds.
./manage-app.sh status 8083 Show whether the tracked app process for that port is running, plus PID, log file, and dashboard URL.
./manage-app.sh logs 8083 Tail the Uvicorn/application log for that port.
./manage-app.sh restart 8083 Stop the tracked process, then start it again with the same setup and health-check behavior.
./manage-app.sh stop 8083 Stop the tracked app process and remove the PID file.
./manage-app.sh seed gc_live.db Seed the selected SQLite database with the demo fleet history.

Ingestion Pipeline Architecture

The analyzer utilizes a push/pull hybrid ingestion pipeline. It schedules periodic checks across all configured broker nodes, scrapes newly appended GC log bytes, parses events, and commits aggregated metrics.

Scraping Modes

The collection layer supports two scraper paradigms configured per-node:

  • Local Scraper (source: local): Reads log files locally on disk. Uses file seeks and offsets mapped to file paths to read only newly appended log data.
  • SSH Scraper (source: ssh): Connects remotely via Paramiko SSH. To optimize network load and memory consumption, the collector queries remote files' inodes and sizes using: 
    stat -c '%i %s' '/path/to/log' 2>/dev/null || stat -f '%i %z' '/path/to/log'
    If the remote file's inode matches the recorded state and its size is greater than or equal to the previous offset, the analyzer executes: 
    tail -c +[OFFSET] '/path/to/log'
    This fetches only the delta block since the last scrape tick, keeping bandwidth usage low.

Storage Layer

The system relies on an embedded SQLite database (gc_history.db or gc_live.db) to track cluster topology metadata, hourly performance metrics, and incremental collector states.

Schema Design

Table Primary Key Columns Role
instances id region, env, cluster, grp, role, idx, heap_max_mb, collector Fleet inventory registry. Tracks JVM details per cluster.
metrics (instance_id, ts) heap_used_mb, heap_max_mb, heap_after_pct, pause_avg_ms, pause_p99_ms, pause_max_ms, full_gc_count, young_count, gc_per_min, time_in_gc_pct, throughput_pct Stores collected metrics aggregated by collection interval ticks. Powers trends.
collector_state (instance_id, file_path) inode, offset, updated_ts Remembers the byte offset and inode of local or remote GC logs. Prevents duplicate scraping.

Aggregations Optimization

Database-Side Query Performance

To display 30-day historical charts, the application aggregates metric records to a daily resolution. Previously calculated in Python memory, this query is now executed on the SQLite side using a mathematical grouping: 

SELECT (ts / 86400) * 86400 AS day, AVG(heap_used_mb) AS heap_used_avg, MAX(heap_used_mb) ... GROUP BY day

Codebase Component Breakdown

The Python packages and web resources are organized logically as follows:

1. Server & API Controller (gcanalyzer/app.py)

An asynchronous FastAPI application hosting API endpoints. It runs an event-loop background task that executes cluster scheduler ticks, handles authentication via the `gc_session` cookie, and exposes JSON APIs for topology, snapshots, trends, and job history.

2. Scraper Task Scheduler (gcanalyzer/scheduler.py)

Uses a thread pool executor to concurrently query remote or local nodes. It feeds log text to the parser and writes offsets and resulting statistics back to the database in a sequential transaction block. The concurrency limit is read from `config.json`.

3. Log Collector (gcanalyzer/collector.py)

Hosts Paramiko SSH client connections and local filesystem access functions. Contains the logic for the remote delta-syncing strategy (utilizing remote tailing) and maps default Kafka GC log globs based on node roles (broker, controller, history, zookeeper).

4. Log Parser (gcanalyzer/parser.py)

Tokenizes unified GC logs (e.g., standard JDK 11+ `-Xlog:gc*` formatting) line-by-line using regular expressions to capture timestamps, garbage collection phases, heap transitions (occupied vs limit), and pause times.

5. Fleet Health Analyzer (gcanalyzer/analyzer.py)

Scores metrics and assigns an academic grade (A-F) based on thresholds for GC throughput, time-in-GC percentage, pause averages, and Full GC occurrences. Also generates tuning advice (e.g., advising G1 upgrades, GCTimeRatio modifications, or Metaspace limits).

Onboarding Clusters

New Kafka environments are registered with the application by submitting a configuration file in YAML format. The system supports full customization of authentication credentials and log paths.

YAML Configuration Schema

Below is a fully populated template schema detailing all available options:

cluster: demo-prod-cluster     # Unique identifier
region: AMER                   # Grouping tier
env: PROD                      # Environment tier (PROD, UAT, etc.)

defaults:
  source: ssh                  # ssh | local
  port: 22
  user: kafka-admin
  key_path: ~/.ssh/id_rsa      # Private key path
  kafka_home: /opt/kafka       # Resolves default log directory
  # password: secret           # Or fallback password if key is not used

nodes:
  - id: broker-1
    role: broker
    host: 10.0.1.10
  - id: broker-2
    role: broker
    host: 10.0.1.11
  - id: controller-1
    role: controller
    host: 10.0.1.20
    # Override paths explicitly if defaults don't fit
    log_paths:
      - /var/log/kafka/gc-server.log*
  - id: zk-1
    role: zookeeper
    source: local
    local_paths:
      - ./samples/zookeeper-1-gc.log
Name Duplicate Validation

To prevent accidentally overwriting configurations, onboarding a new cluster whose name matches an already existing cluster YAML file returns an API 400 Bad Request error. If you wish to update a cluster, edit the configuration in the Settings panel directly.

Dashboard Reference

The UI operates as a Single-Page Application (SPA) providing multi-tier navigation views.

Navigation Trees

The sidebar renders a structured topology: Region → Environment → Cluster → Component Group → JVM Instance. Clicking a tier displays a corresponding context panel:

  • Fleet View: Displays a grid of all registered clusters showing aggregated health and active alert counts.
  • Cluster Overview: Shows total aggregate memory usage, JVM version configurations, active telemetry (e.g. worst pause time in 24h), and a targeted "Needs Attention" watch list highlighting degraded nodes.
  • Instance Detail: Shows current health scores, recommendations, active alerts, recent charts, and 30-day trend lines.

Alerting & Diagnostic Logic

The analyzer evaluates statistics from the last hour of records to identify operational issues.

Alert Categories

Alert Type Severity Trigger Metric Condition Implication
Full GC Critical full_gc_count > 0 A Full Stop-The-World GC took place. Indicates heap exhaustion.
Long Pause Critical pause_max_ms > 500ms A single STW pause exceeded 500ms. Likely to cause Kafka broker timeout drops.
Heap Pressure Warning heap_after_pct > 85% Live set after collection occupies more than 85% of total heap. High risk of OOM.
GC Storm Warning time_in_gc_pct > max(5%, 3x baseline) The JVM is spending significant CPU resources running GC cycles.
GC Frequency Warning gc_per_min > max(20, 2x baseline) High allocation rates causing rapid GC cycles. Degrades application throughput.

Administration & Troubleshooting

The backend serves static files and handles scheduler tasks in the background. Use manage-app.sh status 8083, manage-app.sh logs 8083, and manage-app.sh stop 8083 for routine process control before dropping down to module-level commands.

Managing Concurrency

If the analyzer is scraping hundreds of hosts, adjust the maximum parallel scraping worker limit in the root `config.json` file: 

{
  "concurrency": 25
}

Scraping Job Logs Pop-up

Clicking **"View Logs"** in the jobs sidebar opens an asynchronous log modal. You can inspect connection attempts, files found, read byte sizes, parsing stats, and errors (such as SSH timeout or permission denials) in real-time.

Running Scrapes Manually

You can force a single scraping pass to run outside of the FastAPI runtime. Execute the following command from the project root directory: 

.venv/bin/python -m gcanalyzer.scheduler --db gc_live.db --once
